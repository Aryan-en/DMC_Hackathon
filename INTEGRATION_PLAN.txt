3.1 MEA (MINISTRY OF EXTERNAL AFFAIRS)
────────────────────────────────────────────────────────────────────────────

SOURCE: https://www.mea.gov.in/foreign-relations.htm

DATA EXTRACTION APPROACH:
  Method: Web scraping (BeautifulSoup4) + Manual API integration
  
  KEY DATA POINTS TO EXTRACT:
    • Bilateral relations with each country
    • Diplomatic statements & press releases
    • Trade & economic statistics
    • Cultural exchanges
    • Defense & security partnerships
    • treaties & agreements
    
  SCRAPING TARGETS:
    • /foreign-relations.htm - Country relations index
    • /foregin-relations/[country]/ - Bilateral relation pages
    • /sites/default/files/statements/ - Press releases
    
  DATA SCHEMA (Proposed):
    {
      "country": "string (e.g., 'China', 'United States')",
      "relation_type": "enum['bilateral', 'multilateral', 'disputed']",
      "status": "enum['stable', 'tense', 'active_dispute', 'conflict']",
      "trade_volume": "float (USD millions)",
      "agreements": ["list of agreements"],
      "last_summit": "ISO date",
      "key_issues": ["list of current issues"],
      "sentiment": "enum['positive', 'neutral', 'negative']",
      "confidence_score": "float (0-1)",
      "last_updated": "ISO timestamp",
      "source": "MEA"
    }

IMPLEMENTATION DETAILS:
  • Create /backend/ingestors/mea_scraper.py
  • Run scheduled scrape weekly (APScheduler)
  • Normalize country names to ISO 3166 codes
  • Store raw data in Bronze layer (Delta Lake)
  • Process & enrich in Silver layer
  
SAMPLE CODE STRUCTURE:
  
  from bs4 import BeautifulSoup
  import requests
  from datetime import datetime
  
  def fetch_mea_country_data(country_code):
      url = f"https://www.mea.gov.in/foreign-relations.htm"
      response = requests.get(url)
      soup = BeautifulSoup(response.content, 'html.parser')
      
      # Extract country relations data
      relations = {
          "country": country_code,
          "bilateral_data": extract_bilateral_info(soup),
          "trade_volume": extract_trade_stats(soup),
          "agreements": extract_agreements(soup),
          "sentiment": analyze_sentiment(soup),
          "timestamp": datetime.utcnow().isoformat()
      }
      
      return relations


3.2 WORLD BANK INDICATORS (via IndiAPIs)
────────────────────────────────────────────────────────────────────────────

SOURCE: https://data.worldbank.org (IndiAPIs wrapper)

APPROACH: Direct REST API calls to World Bank API

KEY INDICATORS TO COLLECT (India + Comparative):
  
  ECONOMIC:
    • GDP (NY.GDP.MKTP.CD)
    • GDP Growth Rate (NY.GDP.MKTP.KD.ZS)
    • Inflation, Consumer Prices (FP.CPI.TOTL.ZG)
    • Foreign Direct Investment (BX.KLT.DINV.CD.WD)
    • Trade Volume (NE.EXP.GNFS.CD + NE.IMP.GNFS.CD)
    
  DEMOGRAPHIC:
    • Population (SP.POP.TOTL)
    • Urban Population (SP.URB.TOTL)
    • Life Expectancy (SP.DYN.LE00.IN)
    • Literacy Rate (SE.ADT.LITR.ZS)
    
  DEVELOPMENT:
    • Healthcare Access (SH.XPD.CHEX.GD.ZS)
    • School Enrollment (SE.ENR.PRIM.FM.ZS)
    • Unemployment (SL.UEM.TOTL.ZS)
    
  DATA SCHEMA (Proposed):
    {
      "country": "string",
      "indicator_code": "string (e.g., 'NY.GDP.MKTP.CD')",
      "indicator_name": "string",
      "value": "float",
      "year": "int",
      "unit": "string",
      "source": "World Bank",
      "last_updated": "ISO timestamp"
    }

IMPLEMENTATION DETAILS:
  • Use direct World Bank API (https://api.worldbank.org/v2/)
  • Create /backend/ingestors/worldbank_fetcher.py
  • Fetch data monthly (first day of month)
  • Support historical data retrieval (10-year lookback)
  • Cache results in PostgreSQL
  
SAMPLE CODE STRUCTURE:
  
  import requests
  from datetime import datetime
  
  WORLD_BANK_API = "https://api.worldbank.org/v2"
  
  def fetch_worldbank_indicator(country_code, indicator_code):
      url = f"{WORLD_BANK_API}/country/{country_code}/indicator/{indicator_code}?format=json&per_page=100"
      response = requests.get(url)
      data = response.json()
      
      indicators = []
      for record in data[1]:
          indicators.append({
              "country": country_code,
              "indicator_code": indicator_code,
              "indicator_name": data[0][0]['indicator']['value'],
              "value": float(record['value']) if record['value'] else None,
              "year": int(record['date']),
              "timestamp": datetime.utcnow().isoformat()
          })
      
      return indicators

NEO4J SCHEMA (Proposed):
  
  Nodes:

    • CORRELATES (strength: 0-100)
    • RESTRICTS (domain: e.g., 'trade', 'security')
    • REGULATES (domain, strength)
    • SHAPES (domain, strength)

API ENDPOINTS:
  GET /api/knowledge-graph/nodes

  ├─ Check data quality constraints
  ├─ Identify conflicts/duplicates
  └─ Flag for manual review

  ├─ Compute graph centrality metrics

4.8 SECURITY & GOVERNANCE (/security)
────────────────────────────────────────────────────────────────────────────

PURPOSE:
  Monitor zero-trust architecture, access control, audit logs, and
  data classification compliance.

DATA REQUIREMENTS:
  • Audit logs (access, exports, config changes)
  • Policy violations trend (7-day history)
  • User clearance levels
  • Data lineage with classification
  
SECURITY DATA MODEL:
  
  Access Control:
    • Users belong to Groups (roles)
    • Groups have Permissions on Resources
    • Permissions: READ, WRITE, EXPORT, DELETE, ADMIN
    • Classifications: UNCLASS, FOUO, SECRET, TOP SECRET, TS/SCI
    • Users have clearance_level matching classification
    • ABAC (Attribute-Based Access Control):
      - User attributes: country, dept, clearance
      - Resource attributes: classification, domain, tier
      - Policy: User.clearance >= Resource.classification
  
  Audit Log Schema:
    {
      timestamp: ISO datetime,
      user_id: string,
      action: enum[QUERY, READ, WRITE, EXPORT, DELETE, CONFIG_CHANGE],
      resource: string (table/dataset/api),
      classification: enum[UNCLASS, FOUO, SECRET, TS],
      status: enum[ALLOW, DENY],
      ip_address: string,
      user_agent: string,
      details: {request_params, response_size, execution_time}
    }
  
  Data Lineage:
    {
      dataset_id: string,
      source: [dataset_ids],
      transforms: [{name, type, sql/code}],
      consumers: [{application, frequency, access_level}],
      classification: enum,
      owner: user_id,
      last_accessed: timestamp
    }

API ENDPOINTS:
  GET /api/security/audit-log
    └─ Params: {from_date, to_date, user_id, action, status}
    └─ Returns: {logs: [{...}], total_count: int}
    └─ Data Source: PostgreSQL audit table
    └─ Query Limit: 10K records
    
  GET /api/security/violations-trend
    └─ Returns: {trend: [{day, violation_count, warning_count}]}
    └─ Data Source: Audit log analysis
    └─ Period: Last 7 days
    
  GET /api/security/user-clearance
    └─ Returns: {user_id, clearance_level, groups, resources_accessible}
    └─ Data Source: User-Group-Permission joins
    └─ Update Frequency: Cache every 5 minutes
    
  GET /api/security/data-lineage/:dataset
    └─ Returns: {sources: [...], transforms: [...], consumers: [...]}
    └─ Data Source: Query metadata + manual annotations
    └─ DAG visualization
    
  POST /api/security/export-request
    └─ Input: {dataset, format, classification_assurance}
    └─ Returns: {request_id, review_status, expiration_time}
    └─ Workflow: Requires approval if data is SECRET+
    
  POST /api/security/access-check
    └─ Input: {user_id, resource, action}
    └─ Returns: {allowed: boolean, reason: string}
    └─ Real-time access control decision

CLASSIFICATION LEVELS:
  • UNCLASSIFIED - Public information (no restriction)
  • FOUO (For Official Use Only) - Internal use only
  • SECRET - Limited distribution, specific clearance required
  • TOP SECRET - Highly restricted, SCI need-to-know
  • TS/SCI - Top Secret with Sensitive Compartmented Information

7.1 MEA DATA INGESTION PIPELINE
────────────────────────────────────────────────────────────────────────────

Trigger: Weekly schedule (Sunday 00:00 UTC)

Step 1: Scrape MEA Website
  └─ mea_scraper.py → BeautifulSoup
     Target URLs:
       • https://www.mea.gov.in/foreign-relations.htm (main page)
       • https://www.mea.gov.in/foreign-relations/[country]/
       • Parse HTML tables, extract country relations data
       Output: Raw JSON documents
       
Step 2: Validate & Normalize
  └─ Normalize country names to ISO 3166 codes
     Validate data types, ranges
     Extract structured fields (trade volume, agreements, etc.)
     Output: Validated JSON
     
Step 3: Publish to Kafka
  └─ Topic: mea.relations.raw
     Key: country_code (for partitioning)
     Value: {country, relation_type, status, ...}
     Partitions: 12 (one per major region)
     
Step 4: Consume & Process (NLP)
  └─ Consumer: NLP Pipeline Pod
     Operations:
       • Extract sentiment from diplomatic language
       • Named entity recognition (countries, officials, events)
       • Relationship extraction (agreements, disputes)
       • Classification (hostile, stable, cooperative)
     Output: Enriched triplets
     
Step 5: Store in Knowledge Graph
  └─ Neo4j Write:
     Create nodes: Country (if not exists)
     Create relationships: BILATERAL_RELATION, HAS_AGREEMENT, etc.
     Update properties: sentiment, last_updated, confidence
     
Step 6: Archive Raw Data
  └─ Delta Lake Bronze Layer
     Path: s3://ontora-datalake/bronze/mea/relations/[year]/[month]/[day]

7.3 NLP PROCESSING PIPELINE (Real-Time)
────────────────────────────────────────────────────────────────────────────

Trigger: Streaming (Kafka topic consumption)

Topics consumed: mea.relations.raw, news.articles.global, social.sentiment.twitter

Step 1: Language Detection
  └─ Input: Raw text
     Method: fasttext langdetect
     Output: language code (en, zh, ar, etc.)
     
Step 2: Entity Recognition (spaCy)
  └─ Model: spacy.blank('en').add_pipe(...) for each language
     Output: [{text, label, start, end}]
     Labels: PERSON, ORG, GPE (country), EVENT, DATE, MONEY
     
Step 3: Dependency Parsing & Relationship Extraction
  └─ spaCy: .sents, .token_dependencies
     Pattern matching for relationships:
       • "Country A signed agreement X with Country B"
       • "Person P is minister of Country C"
     Output: {subject, verb, object}
     
Step 4: LLM Classification (Optional - LLaMA-3)
  └─ For complex cases, use LLM for:
       • Entity type refinement
       • Sentiment assessment
       • Confidence scoring
     Prompt: "Classify this entity: [text]"
     Output: {class, confidence: 0-1}
     
Step 5: Validation & Conflict Detection
  └─ Check against existing KG:
       • Is this relationship new?
       • Does it conflict with existing data?
       • What is confidence/credibility?
     
Step 6: Publish Triplets to Kafka
  └─ Topic: kg.triplets.extracted
     Key: source_document_id
     Value: [{subject, predicate, object, confidence, source_doc}]
     
Step 7: Store in Neo4j
  └─ Merge logic (idempotent):
     MERGE (s:Entity {id: subject_id})
     MERGE (o:Entity {id: object_id})
     MERGE (s)-[r:RELATIONSHIP {type: predicate}]->(o)
     SET r.confidence = ..., r.sources = [...]
     
Step 8: Archive to Delta Lake Bronze
  └─ Raw documents + metadata
     Path: s3://ontora-datalake/bronze/nlp/documents/[date]/
     Retention: 7 days

Pseudo-code for NLP Pipeline:

  async def process_document(doc: Document):
      # Detect language
      lang = detect_language(doc.text)
      
      # NLP processing
      nlp = spacy.load(f"{lang}_core_web_lg")
      doc_obj = nlp(doc.text)
      
      entities = [
          {"text": ent.text, "label": ent.label_, "start": ent.start_char}
          for ent in doc_obj.ents
      ]
      
      # Relationship extraction
      triplets = []
      for sent in doc_obj.sents:
          rels = extract_relations(sent)  # Custom function
          triplets.extend(rels)
      
      # Validate
      validated = validate_triplets(triplets, kg_client)
      
      # Publish & store
      await kafka_producer.send("kg.triplets.extracted", validated)
      await neo4j_client.write_triplets(validated)
      
      return {"entities": entities, "triplets": triplets}

8. SECURITY & GOVERNANCE FRAMEWORK
================================================================================

8.1 AUTHENTICATION
────────────────────────────────────────────────────────────────────────────

Method: OAuth2 + JWT + MFA

Flow:
  1. User submits credentials (username/password)
  2. Backend validates against user database
  3. Requires MFA (TOTP or hardware key)
  4. Backend issues JWT tokens:
     - Access token (15-minute expiry)
     - Refresh token (30-day expiry)
  5. Client stores tokens in httpOnly cookies
  6. Each API call includes: Authorization: Bearer <token>
  7. Backend validates JWT signature & claims


8.2 AUTHORIZATION (RBAC + ABAC)
────────────────────────────────────────────────────────────────────────────

Role-Based Access Control (RBAC):
  
  Roles:
    • Administrator: Full system access
    • Analyst: Read all data, can create reports, cannot delete
    • Operator: Monitor dashboards only, read-limited access
    • Guest: Read-only public data
  
  Role → Permission Mapping:
    • Administrator: [READ, WRITE, DELETE, ADMIN, EXPORT]
    • Analyst: [READ, WRITE, EXPORT]
    • Operator: [READ]
    • Guest: [READ_PUBLIC]

Attribute-Based Access Control (ABAC):
  
  Data is tagged with classification:
    {classification: UNCLASS | FOUO | SECRET | TS | TS/SCI}
  
  Users have clearance:
    {clearance_level: UNCLASS | FOUO | SECRET | TS | TS/SCI}
  
  Access Rule:
    User.role permits action?
    AND User.clearance >= Resource.classification?
    AND User.department in Resource.authorized_departments?
    → ALLOW / DENY


8.3 DATA CLASSIFICATION
────────────────────────────────────────────────────────────────────────────

Classification Scheme:

  UNCLASSIFIED (U//)
    • Public information
    • No access restrictions
    • Can be freely shared
  
  FOR OFFICIAL USE ONLY (FOUO)
    • Internal government use only
    • Not for public distribution
    • Some access logging required
  
  SECRET (S//)
    • Restricted distribution
    • Need-to-know basis
    • Specific clearance required
    • Access fully logged
    • Cannot be exported without approval
  
  TOP SECRET (TS//)
    • Highly restricted
    • Specific need-to-know
    • TS clearance required
    • All access logged & monitored
    • Audit trail mandatory
  
  TOP SECRET / SCI (TS//SCI)
    • Compartmented information
    • Strictest control
    • Need-to-know + compartment approval
    • Full monitoring, audit, approval for access

Data Classification Decision Tree:
  
  Q: Is this publicly available info?
  └─ YES → UNCLASSIFIED
  └─ NO → Q2
  
  Q2: Could disclosure harm government interests?
  └─ NO → FOUO
  └─ YES → Q3
  
  Q3: Could disclosure cause "serious damage" to national security?
  └─ NO → SECRET
  └─ YES → Q4
  
  Q4: Does this contain compartmented/sensitive intelligence?
  └─ NO → TOP SECRET
  └─ YES → TOP SECRET / SCI


8.4 DATA GOVERNANCE POLICIES
────────────────────────────────────────────────────────────────────────────

Data Owner Responsibilities:
  • Classify data appropriately
  • Define retention period
  • Authorize users/departments
  • Review access logs regularly
  • Ensure data quality
  
Data Consumer Responsibilities:
  • Use data only for authorized purpose
  • Protect data according to classification
  • Report unauthorized access
  • Request deletion after use
  
Technical Controls:
  • Encryption at rest (AES-256)
  • Encryption in transit (TLS 1.3)
  • Database-level access control (row/column level)
  • Audit logging (every read/write/export)
  • Data masking (PII, sensitive values)
  • Anonymization (where applicable)
  
Compliance Monitoring:
  • Quarterly compliance audits
  • Annual security assessments
  • Continuous monitoring (SIEM)
  • Incident response playbooks
